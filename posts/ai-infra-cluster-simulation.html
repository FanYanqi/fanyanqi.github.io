01 / 研究范围问题定义：为什么需要仿真

推理引擎的每一次调度策略改动——连续批处理窗口、prefill/decode 分离、KV cache 淘汰——若要在真实集群上验证， 需要固定数量的 GPU、稳定的网络环境、可复现的请求 trace，单次实验动辄消耗数十 GPU 小时。 仿真把这条链路搬上 CPU，用性能模型 + 离散事件调度替代真实前向计算。

研究的三个子问题SCOPE

(1) 性能预测：给定模型、集群拓扑、请求负载，预测吞吐 / TTFT / TPOT / 尾延迟。 (2) 策略评估：对比不同调度、批处理、并行策略的端到端效果，无需占用真实 GPU。 (3) 设计空间探索：在硬件未到位前评估 GPU 型号、网络拓扑、显存层级的影响。

核心矛盾 真实集群测量 准确但昂贵且不可复现；纯解析模型 廉价但失真。 2024–2026 年的工作正在把两者缝合：用真实引擎的调度器代码 + 一次小规模 profiling 数据，驱动一个在 CPU 上跑的离散事件仿真器，达到 <6% 的端到端误差。

03 / 三层保真度仿真方法的分类谱系

所有仿真器都可归入三层保真度谱系。保真度越高，单次仿真越慢、所需 profiling 数据越多； 选择哪一层取决于研究问题——容量规划用解析层，调度策略用 DES 层，kernel 优化用 cycle 层。

Tier 1 · 解析 / 回归

快但粗

用闭式公式或回归模型预测延迟。一次 profiling 拟合参数，之后零成本查询。适合大规模设计空间初筛。

LogGPSα-β 模型Dooly 回归APEX+ 解析
Tier 2 · 离散事件

中庸之道

事件驱动的调度器-批次-引擎循环仿真，复用真实引擎调度代码，用 profile 数据替代前向计算。当前主流。

SplitwiseSimFrontierDoolySimLLM-EmuASTRA-Sim
Tier 3 · Cycle / Kernel 级

慢但精

逐 kernel、甚至 cache-line 粒度建模 GPU 执行。用于硬件协同设计、kernel 级优化验证。

LLMServingSimKernelSight-LMASTRA-Sim 3.0Mosaic
2026 年的趋势 工具正在向 混合保真 演进：同一框架内可切换层级。Frontier 在 CPU 上跑 DES，但 kernel 延迟来自校准模型；ASTRA-Sim 3.0 在 cycle-level GPU 模型外套解析网络后端。研究者按需为不同子模块选不同保真度。

这 8 个工具是本报告的核心。它们都在 2024 年后出现，且明确以 vLLM 或同类推理引擎的调度语义为仿真对象。

07 / 推荐架构为 vLLM/SGLang 构建仿真环境的分层方案

综合现有工具的能力，给出一个分层架构：底层复用 ASTRA-Sim 的网络/GPU 模型，中间层复用 vLLM 真实调度器，上层用 profile 数据驱动。

关键设计决策：L1 层不重新实现调度器，而是直接 import vLLM/SGLang 的调度代码（DoolySim、LLM-Emu 已验证此路径）。 L2 层的 kernel 延迟来自一次性 profiling（Dooly 的 taint propagation 可把 profiling 成本降 56%）。 L3 层的网络/GPU 模型复用 ASTRA-Sim 的后端。这样仿真器与真实引擎同构，引擎升级时仿真器只需跟进调度代码差异。

09 / 选型建议按研究目标选择仿真器

没有一个工具覆盖所有需求。以下按四类常见研究目标给出推荐路径。

[ A ]

调度策略 / 批处理算法研究

需要复现连续批处理、chunked prefill、prefill/decode 分离的端到端效果。

  • 首选 Frontier：vLLM 风格，PDD 支持，<4% 误差
  • 或 SplitwiseSim：Azure trace 驱动，Ray 并行扫描
  • 对比基线：真实 vLLM 小规模部署
[ B ]

并行策略 / 集群配置探索

给定模型和集群，找最优 TP/PP/DP/EP 组合与 GPU 型号选型。

  • 首选 APEX+：已验证 vLLM + SGLang，CPU 15 分钟出计划
  • 大规模网络：配 ASTRA-Sim 3.0 的 InfraGraph
  • 能耗优化：APEX+ 内置能耗模型
[ C ]

引擎开发 / 省 GPU 小时

开发阶段预测修改的影响，不占用真实 GPU。

  • 首选 LLM-Emu 或 DoolySim：复用 vLLM 真实调度代码
  • profiling 一次 → 回归模型驱动，Dooly 降 56% profiling 成本
  • kernel 级改动：用 KernelSight-LM
[ D ]

Agent / 多轮工作负载

研究多轮 agent 的 KV cache 跨轮放置、工具调用间隙、session 路由。

  • 首选 AGENTSERVESIM：唯一面向多轮 agent serving
  • 异构/MoE：配 LLMServingSim 2.0
  • 注意：vLLM/SGLang 本身的多轮支持仍在演进，仿真需跟进
落地建议 若从零搭建，推荐路径：(1) 用 Dooly/LLM-Emu 的 profiler 在 1–2 张 GPU 上采集一次 kernel 延迟； (2) 把 profile 数据喂给 Frontier 或 DoolySim 的 DES 内核（复用 vLLM 调度代码）； (3) 网络部分接 ASTRA-Sim 的 analytical 后端做大规模拓扑扩展； (4) 用 APEX+ 做并行计划初筛，再在仿真器里精评。 全流程除步骤 (1) 外不需要 GPU。
