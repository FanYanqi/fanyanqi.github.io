为 vLLM 与 SGLang 构建不依赖大规模集群的推理仿真研究环境

2026-07-19 · 技术调研报告 · 范围: LLM 推理集群仿真

大模型推理引擎的调度、批处理与显存策略研究通常需要 数十张 GPU 才能获得可信的端到端数据。本报告系统梳理 2020–2026 年间的集群仿真器与推理性能建模工作，给出在普通 CPU 上即可复现真实引擎行为（吞吐/延迟误差 < 6%）的技术路径。

覆盖： 2020–2026 · 13 个仿真器 + 7 篇基础 serving 论文 · 数据来源 arXiv / GitHub / ISPASS / ISCA / OSDI / ASPLOS / MLSys / SOSP。

01 · 问题定义：为什么需要仿真

推理引擎的每一次调度策略改动——连续批处理窗口、prefill/decode 分离、KV cache 淘汰——若要在真实集群上验证，需要固定数量的 GPU、稳定的网络环境、可复现的请求 trace，单次实验动辄消耗数十 GPU 小时。仿真把这条链路搬上 CPU，用性能模型 + 离散事件调度替代真实前向计算。

研究的三个子问题

  • 性能预测：给定模型、集群拓扑、请求负载，预测吞吐 / TTFT / TPOT / 尾延迟。
  • 策略评估：对比不同调度、批处理、并行策略的端到端效果，无需占用真实 GPU。
  • 设计空间探索：在硬件未到位前评估 GPU 型号、网络拓扑、显存层级的影响。
核心矛盾： 真实集群测量准确但昂贵且不可复现；纯解析模型廉价但失真。2024–2026 年的工作正在把两者缝合：用真实引擎的调度器代码 + 一次小规模 profiling 数据，驱动一个在 CPU 上跑的离散事件仿真器，达到 <6% 的端到端误差。

03 · 三层保真度

所有仿真器都可归入三层保真度谱系。保真度越高，单次仿真越慢、所需 profiling 数据越多；选择哪一层取决于研究问题——容量规划用解析层，调度策略用 DES 层，kernel 优化用 cycle 层。

Tier 1 · 解析 / 回归 — 快但粗

用闭式公式或回归模型预测延迟。一次 profiling 拟合参数，之后零成本查询。适合大规模设计空间初筛。代表：LogGPS、α-β 模型、Dooly 回归、APEX+ 解析。

Tier 2 · 离散事件 — 中庸之道

事件驱动的调度器-批次-引擎循环仿真，复用真实引擎调度代码，用 profile 数据替代前向计算。当前主流。代表：SplitwiseSim、Frontier、DoolySim、LLM-Emu、ASTRA-Sim。

Tier 3 · Cycle / Kernel 级 — 慢但精

逐 kernel、甚至 cache-line 粒度建模 GPU 执行。用于硬件协同设计、kernel 级优化验证。代表：LLMServingSim、KernelSight-LM、ASTRA-Sim 3.0、Mosaic。

2026 年的趋势： 工具正在向混合保真演进：同一框架内可切换层级。Frontier 在 CPU 上跑 DES，但 kernel 延迟来自校准模型；ASTRA-Sim 3.0 在 cycle-level GPU 模型外套解析网络后端。研究者按需为不同子模块选不同保真度。

04 · 推理仿真器：直接对标 vLLM / SGLang 的工具

这 8 个工具是本报告的核心。它们都在 2024 年后出现，且明确以 vLLM 或同类推理引擎的调度语义为仿真对象。

LLMServingSim 2.0 (KAIST, ISPASS'26)

Cycle-level 仿真器，Python 前端直接镜像 vLLM 的连续批处理调度器，后端接 ASTRA-Sim 的 C++ 解析网络层。延迟数据由基于 vLLM 的逐层 profiler 采集。统一研究异构加速器、CXL/PIM 分离内存、MoE 路由与 TP/PP/EP/DP 多实例并行。
github · 保真度: cycle-level · 与 vLLM 调度器同构

SplitwiseSim (Microsoft Research, ISCA'24)

离散事件仿真器，Hydra+YAML 配置，基于 Azure 生产 trace 生成请求。性能模型来自实测 perf-model.csv 的线性预测器。支持 ORCA 实例、Splitwise 阶段分离、连续批处理迭代级调度，可用 Ray 并行扫描大规模配置空间。
github · 保真度: DES · 面向 DGX-A100/H100 集群

APEX+ (Microsoft, arXiv'24)

给定 LLM + 设备集群 + 请求，自动生成最优并行执行计划 (TP/PP/DP)。建模迭代级批处理，利用 LLM 重复结构缩减设计空间，可扩展至万亿参数。README 明确含 "Validating APEX+ with vLLM and SGLang" 章节——直接对标这两个引擎。
github · arXiv 2411.17651 · CPU 上 15 分钟找到最优计划 · 比启发式快 3.37× · 能耗降 45%

Frontier (CUHK NetX-lab, arXiv'26)

离散事件仿真器，仿真 vLLM 风格服务行为，把 CUDA Graph、投机解码/MTP、prefix caching、量化、chunked prefill、分层缓存作为真实运行时行为而非简单加速因子。支持 co-located 与 Prefill-Decode 分离 (PDD)。纯 CPU 端到端仿真，可扩展到 1K+ GPU。
github · arXiv 2605.21312 · 16×H800 吞吐误差 <4% · PDD 延迟误差从 44.9% 降至 6.4%

DoolySim / Dooly (arXiv'26)

复用真实 vLLM 调度代码使批处理决策与生产一致；用 taint propagation 标注算子输入维度来源，仅对延迟库缺失算子做选择性 profiling，消除跨配置重复 profiling。Random Forest 回归预测每步延迟，作为即插即用后端。
github · arXiv 2605.07985 · TTFT 5% MAPE · TPOT 8% MAPE · profiling GPU 小时减少 56.4%

LLM-Emu (arXiv'26)

面向 vLLM 的 serving-native emulator，保留生产 HTTP / 调度 / KV-cache / 输出处理路径，仅用 profile 采样的延迟与合成 token 替换 GPU 前向执行。不重新实现调度器，直接复用 vLLM 真实调度逻辑。
arXiv 2605.00616 · TPOT/ITL 误差 ≤4.8% · E2E 延迟 ≤5.3% · 吞吐 ≤1.9%

KernelSight-LM (arXiv'26)

细粒度 token 级推理仿真器，把每个 serving step 分解为 roofline kernel 模型 + 学习效率项 + 通信模型 + host 开销，通过离散事件调度器组合并捕获 prefix caching、continuous batching。两个预测层级：cross-generation（无需目标 GPU 实测）与 target-measured。
arXiv 2606.28565 · target-measured per-kernel 3.8% 误差 · 端到端 TPOT 6.2%

AGENTSERVESIM (arXiv'26)

硬件感知的多轮 LLM agent 服务仿真器，以程序粒度评估服务策略。可组合模块：Program Orchestrator、Tool Simulator、Session-Aware Router、KV Residency Model（跨 HBM/DRAM/CXL/eviction 跟踪 KV 放置）。填补现有仿真器只处理无状态请求、忽略多轮 agent 动态的空白。
arXiv 2606.09613 · 关键指标误差 <6% · 纯 commodity CPU 运行

05 · 训练仿真器：可迁移到推理的基础设施

训练侧仿真器更早成熟，其网络拓扑建模、集合通信仿真、GPU 执行模型是推理仿真的基础设施。ASTRA-Sim 3.0 已明确把推理列为用例。

ASTRA-Sim 系列 (Georgia Tech, 2020→2026)

开源分布式 ML 仿真器，plug-and-play 的 compute/network/memory 后端架构。网络后端可选 Analytical / ns-3 / Garnet(gem5)，计算后端有 Roofline / SCALE-Sim。2.0 (2023) 引入层次化网络与分离式内存；3.0 (2026) 引入 cache-line 级 GPU 模型与 InfraGraph 基础设施表示，明确支持推理用例。
github · v2 arXiv · v3 arXiv · 该领域最活跃的开源框架

SimAI (阿里巴巴, ~2024)

结合 ns-3 packet-level 网络仿真 + analytical compute model，重点验证大规模 LLM 训练的 collective communication 性能。面向万卡训练 · ns-3 高保真网络。

Mosaic (MLSys'23)

针对分布式 DL 训练的 GPU 集群仿真器，建模 GPU compute kernel、NVLink/IB 通信与 collective 算法，用于快速探索并行策略/拓扑设计的性能空间，比 ns-3 快若干数量级。开源程度有限。

SimPy / ns-3 / CloudSim+ (开源社区)

通用 DES 框架。SimPy 轻量 Python，适合原型级调度/排队仿真；ns-3 是 packet-level 网络仿真器，被 ASTRA-Sim/SimAI 作为可插接网络后端；CloudSim+ 是 CloudSim 的现代 Java fork，AI/ML 负载需自定义 PE/Cloudlet 扩展。均不含 GPU/collective 专用建模。
SimPy · ns-3 · CloudSim+

仿真器不是凭空建模——它们复现的是这些基础论文提出的机制。理解被仿真对象才能选对仿真器。

07 · 推荐架构：为 vLLM/SGLang 构建仿真环境的分层方案

综合现有工具的能力，给出一个分层架构：底层复用 ASTRA-Sim 的网络/GPU 模型，中间层复用 vLLM 真实调度器，上层用 profile 数据驱动。

  • L0 · Workload：请求 trace（Azure 生产 / LMSYS / 自造）→ 变长 prompt + 输出长度分布 + 到达过程。
  • L1 · Engine Scheduler（复用真实代码）：vLLM Scheduler / SGLang Scheduler · 连续批处理 · chunked prefill · prefix cache · 路由。
  • L2 · Performance Model（profile 驱动）：逐层 kernel 延迟（Dooly profiler / KernelSight roofline）+ KV cache 显存模型（PagedAttention 块状）+ 通信模型（α-β / ns-3）。
  • L3 · Infrastructure（ASTRA-Sim 后端）：GPU 拓扑（NVLink/PCIe）· 网络拓扑（RoCE/IB, torus/fat-tree）· 显存层级（HBM/DRAM/CXL）。计算后端 Roofline / SCALE-Sim，网络后端 Analytical / ns-3 / Garnet。
  • L4 · Output：吞吐 / TTFT / TPOT / 尾延迟 / 显存占用 / 能耗。

关键设计决策： L1 层不重新实现调度器，而是直接 import vLLM/SGLang 的调度代码（DoolySim、LLM-Emu 已验证此路径）。L2 层的 kernel 延迟来自一次性 profiling（Dooly 的 taint propagation 可把 profiling 成本降 56%）。L3 层的网络/GPU 模型复用 ASTRA-Sim 的后端。这样仿真器与真实引擎同构，引擎升级时仿真器只需跟进调度代码差异。

08 · 领域演进：2020 → 2026 的关键节点

  • 2020 · ISPASS — ASTRA-Sim 发布：Georgia Tech 提出解耦 compute/communication/collective 三层的 DES 仿真器，奠定该领域基础范式。
  • 2022 · OSDI — Orca：iteration-level batching，提出连续批处理的调度抽象，成为后续所有推理仿真器必须复现的调度语义。
  • 2023 · SOSP — vLLM / PagedAttention：分页式 KV cache 使推理仿真的显存建模从连续变为块状，所有 2024+ 仿真器都围绕此重构。
  • 2023 · ISPASS / MLSys — ASTRA-Sim2.0 与 Mosaic：层次化网络 + 分离式内存建模；GPU kernel + 通信联合建模追求比 ns-3 更快的探索速度。
  • 2024 · OSDI / ISCA — 推理仿真器集中涌现：DistServe、Splitwise 提出 prefill/decode 解耦；SplitwiseSim 作为首个对标 ORCA/Splitwise 的开源推理集群仿真器发布。APEX+ 验证 vLLM + SGLang。
  • 2025 · ASPLOS — vAttention · Parrot (OSDI'24)：CUDA 虚拟内存管理替代 PagedAttention；语义变量暴露跨请求相关性。仿真器需对两种显存管理建模。
  • 2026 · arXiv 集中爆发 — 6+ 个 LLM serving 仿真器发表：Frontier、DoolySim、LLM-Emu、KernelSight-LM、AGENTSERVESIM、LLMServingSim 2.0、ASTRA-Sim 3.0 同年出现。误差降至 <6%，普遍支持纯 CPU 运行，多个直接复用 vLLM 调度代码。

09 · 选型建议：按研究目标选择仿真器

没有一个工具覆盖所有需求。以下按四类常见研究目标给出推荐路径。

[ A ] 调度策略 / 批处理算法研究

需要复现连续批处理、chunked prefill、prefill/decode 分离的端到端效果。

  • 首选 Frontier：vLLM 风格，PDD 支持，<4% 误差
  • 或 SplitwiseSim：Azure trace 驱动，Ray 并行扫描
  • 对比基线：真实 vLLM 小规模部署

[ B ] 并行策略 / 集群配置探索

给定模型和集群，找最优 TP/PP/DP/EP 组合与 GPU 型号选型。

  • 首选 APEX+：已验证 vLLM + SGLang，CPU 15 分钟出计划
  • 大规模网络：配 ASTRA-Sim 3.0 的 InfraGraph
  • 能耗优化：APEX+ 内置能耗模型

[ C ] 引擎开发 / 省 GPU 小时

开发阶段预测修改的影响，不占用真实 GPU。

  • 首选 LLM-Emu 或 DoolySim：复用 vLLM 真实调度代码
  • profiling 一次 → 回归模型驱动，Dooly 降 56% profiling 成本
  • kernel 级改动：用 KernelSight-LM

[ D ] Agent / 多轮工作负载

研究多轮 agent 的 KV cache 跨轮放置、工具调用间隙、session 路由。

  • 首选 AGENTSERVESIM：唯一面向多轮 agent serving
  • 异构/MoE：配 LLMServingSim 2.0
  • 注意：vLLM/SGLang 本身的多轮支持仍在演进，仿真需跟进
落地建议： 若从零搭建，推荐路径：(1) 用 Dooly/LLM-Emu 的 profiler 在 1–2 张 GPU 上采集一次 kernel 延迟；(2) 把 profile 数据喂给 Frontier 或 DoolySim 的 DES 内核（复用 vLLM 调度代码）；(3) 网络部分接 ASTRA-Sim 的 analytical 后端做大规模拓扑扩展；(4) 用 APEX+ 做并行计划初筛，再在仿真器里精评。全流程除步骤 (1) 外不需要 GPU。
